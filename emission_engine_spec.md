# Mobility Emission Estimation Engine — Implementation Spec
### برای اجرای مستقیم در Claude Code — پروژه Bordeaux Urban Digital Twin

---

## ۰. زمینه (Context) — برای Claude Code

پروژه یک Digital Twin شهری بوردو با ساختار Streamlit است (۵ تب: Overview, Data catalog,
Data Model & Ontology, City Dashboard, Data & Resources). `City Dashboard` منطقه‌محور است
(`zones.py`: Bordeaux, Talence, Pessac) و برای هر منطقه یک نقشه‌ی زنده‌ی ترام/اتوبوس +
یک بخش «Historical trends» با ۵ حالت (`app_pages/city_dashboard.py` → دیکشنری `MODES`)
دارد: bike, car, transit (فقط یادداشت), pedestrian (فقط یادداشت), moto (فقط یادداشت).

داده‌ی هیستوریکال ترافیک خودرو از قبل جمع‌آوری می‌شود:
`collectors/traffic_collector.py` → دیتاست `pc_capte_p_histo_jour` (Bordeaux Métropole
Open Data) → یک ردیف به‌ازای هر سنسور در هر روز، ستون کلیدی `comptage_5m` (تعداد خودرو).
فایل‌های نمونه در `data/traffic/car_traffic_<zone>_<timestamp>.csv`.

**این فاز باید از همین داده‌ی موجود استفاده کند — نیازی به کالکتور جدید نیست.**

---

## ۱. هدف این فاز (دقیقاً همین، نه بیشتر)

ساخت یک **Rule-Based Emission Estimation Engine** ساده که:
- ورودی: داده‌ی هیستوریکال ترافیک خودرو که از قبل جمع‌آوری شده (`comptage_5m` روزانه به تفکیک سنسور)
- خروجی: برآورد مقدار انتشار **CO₂، NOx، PM** (یا یک Emission Score ترکیبی) به‌ازای هر سنسور/روز/منطقه
- نمایش: یک بخش جدید در همان `City Dashboard`، بدون تغییر اساسی در معماری فعلی

### ❌ خارج از scope این فاز (به‌صراحت نباید انجام شود)
- هیچ مدل Machine Learning
- هیچ شبیه‌سازی فیزیکی (dispersion, CFD, و مشابه)
- هیچ کالیبراسیون پیچیده یا داده‌ی سرعت واقعی لحظه‌ای
- هیچ تغییر در ساختار تب‌ها یا فایل‌های موجود بیرون از آنچه در بخش ۵ مشخص شده

---

## ۲. داده‌های ورودی مدل (چیزی که واقعاً داریم / نداریم)

| فیلد لازم برای COPERT/EEA کامل | وضعیت در پروژه | تصمیم برای این فاز |
|---|---|---|
| تعداد خودرو (Vehicle Count) | ✅ داریم (`comptage_5m`، روزانه، به‌ازای سنسور) | مستقیم استفاده می‌شود |
| سرعت متوسط (Average Speed) | ❌ نداریم در `pc_capte_p_histo_jour` | **فرض ثابت**: یک مقدار پیش‌فرض سرعت شهری (پیشنهاد: ۳۰ km/h، قابل تنظیم در config) برای همه‌ی سنسورها در این فاز. مستند شود که این یک ساده‌سازی عمدی PoC است. |
| ترکیب ناوگان (Fleet mix: بنزین/دیزل/الکتریک، سال ساخت) | ❌ نداریم | **فرض ثابت**: یک ضریب انتشار میانگین «ناوگان مختلط اروپایی معمولی» به‌جای تفکیک بر اساس نوع خودرو. |
| نوع/عملکرد جاده (urban/arterial/highway) | جزئی — فیلد `type` وجود دارد ولی مقدارش `BOUCLE` است (نوع سنسور، نه نوع جاده) | در این فاز نادیده گرفته می‌شود؛ همه به‌عنوان «جاده‌ی شهری» فرض می‌شوند |
| طول قطعه‌ی جاده (برای گرم به کیلومتر تبدیل کردن) | ❌ نداریم به‌ازای هر سنسور | **واحد محاسبه**: «انتشار به‌ازای هر عبور خودرو در یک بازه‌ی استاندارد ۱ کیلومتری» — یعنی خروجی per-vehicle-pass است، نه per-actual-road-length. این باید در Dashboard به‌وضوح ذکر شود. |

> **جمع‌بندی این بخش:** مدل روی «آنچه واقعاً داریم» (تعداد خودرو در روز، به تفکیک سنسور/منطقه) کار می‌کند
> و برای فاکتورهای گمشده (سرعت، ناوگان، طول جاده) فرض‌های ساده‌ی مستندشده می‌گذارد. این دقیقاً
> همان «Rule-Based + Proof of Concept» است که خواسته شده، نه یک مدل علمی دقیق.

## ۳. فرمول محاسبه (Rule-Based، بدون ML)

```
Emission(pollutant, sensor, day) = VehicleCount(sensor, day) × EF(pollutant) × UNIT_DISTANCE_KM
```

که در آن:
- `VehicleCount(sensor, day)` = مستقیم از ستون `comptage_5m` در CSV موجود
- `EF(pollutant)` = ضریب انتشار میانگین (گرم بر کیلومتر بر خودرو) — از جدول ثابت (بخش ۴)
- `UNIT_DISTANCE_KM` = `1.0` (ثابت، قابل تنظیم در config — معادل فرض «هر خودرو ۱ کیلومتر از این سنسور عبور کرده»)

جمع‌های سطح بالاتر (روزانه برای کل منطقه، یا کل بازه‌ی زمانی) با `groupby` ساده روی خروجی این فرمول به‌دست می‌آیند — نیازی به منطق اضافه نیست.

## ۴. جدول ضرایب انتشار (Emission Factors) — نسخه‌ی مبتنی بر دیتاست‌های خودِ پروژه

⚠️ **این بخش نسبت به نسخه‌ی اول این spec عوض شده** — چون بررسی Ontology پروژه (`data_models.py` →
رابطه‌ی Mobility↔AirQuality) نشون داد که **دو تا دیتاست مرتبط از قبل توی Data Inventory خودت
ثبت شده‌اند** و به‌جای فرض کلی EEA/EMEP، باید مستقیم از این‌ها استفاده کرد:

### ۴.۱ منبع ضریب CO₂ — دیتاست `mob_emissions_1` (خودِ پروژه، دامنه‌ی Mobility)

**این چیه؟** دیتاست رسمی EEA به نام *"Monitoring CO2 emissions from new passenger cars and
vans"* (بر اساس Regulation EU 2019/631) — شامل CO2 اعلام‌شده (WLTP، g/km) به‌ازای هر خودروی
نو ثبت‌شده در فرانسه، با جزئیات نوع سوخت/جرم/مدل.

**نحوه‌ی استفاده در این فاز:**
1. از این دیتاست، میانگین CO2 (g/km) خودروهای ثبت‌شده در فرانسه (آخرین سال موجود) استخراج شود
2. این عدد به‌جای فرض دستی «۱۵۰ g/km» در `EMISSION_FACTORS_G_PER_KM["CO2"]` قرار بگیرد
3. **محدودیت مهم که باید مستند شود:** این عدد مربوط به خودروهای «نو» (Type-Approval، WLTP) است،
   نه میانگین کل ناوگان در حال تردد (که معمولاً بالاتر است، چون خودروهای قدیمی‌تر هم توی خیابون‌اند).
   برای PoC این ساده‌سازی قابل قبول است؛ فقط باید در Dashboard ذکر شود: *"بر اساس میانگین CO2
   خودروهای نو ثبت‌شده (EEA)، نه میانگین کل ناوگان در حال تردد."*

> اگر دیتاست را داری، Claude Code باید مستقیم آن را بخواند و میانگین را حساب کند —
> عدد را از حافظه/گمان وارد نکند.

**⚠️ نکته‌ی عملی مهم:** طبق Data Inventory خودت، این دیتاست فقط یک لینک صفحه‌ی EEA Data Hub
دارد (`Dataset URL`)، نه یک API سبک — و در واقعیت این دیتاست میکروداده‌ی سطح‌اروپا با میلیون‌ها
ردیف است (حجم بالا، دانلود/پردازشش برای یک PoC هزینه‌ی زیادی دارد). **پیشنهاد عملی برای این فاز:**
به‌جای دانلود کامل دیتاست، از آخرین **آمار میانگین رسمی منتشرشده‌ی EEA** برای فرانسه/اروپا استفاده
کن (مثلاً از گزارش‌های خلاصه‌ی سالانه‌ی EEA — عدد را Claude Code باید هنگام اجرا از سایت رسمی EEA
جستجو/تأیید کند، نه از این spec کپی کند چون سالانه تغییر می‌کند). اگر بعداً وقت/نیاز بود، دانلود و
پردازش کامل دیتاست خام می‌تواند فاز ۲ باشد.

### ۴.۲ ضرایب NOx و PM — همچنان از EEA/EMEP Guidebook (Tier 1)

دیتاست `mob_emissions_1` فقط CO2 دارد (NOx/PM جزو Regulation 2019/631 نیست). برای این دو
آلاینده همچنان باید از جدول Tier 1 گاید‌بوک EEA/EMEP استفاده کرد (placeholder بخش ۴.۳ پایین)،
مگر این‌که در دیتاست‌های AirQuality (`bor_inventaire_polluants_1`) ضریب مستقیمی پیدا شود که
معمولاً این‌طور نیست — آن دیتاست خودِ مقدار انتشار سالانه‌ی کل منطقه است، نه ضریب per-km.

### ۴.۳ Placeholder برای NOx/PM (تا زمان تأیید نهایی)

```python
# emission_factors.py
EMISSION_FACTORS_G_PER_KM = {
    "CO2": None,    # ← از dataset mob_emissions_1 محاسبه شود (میانگین فرانسه، آخرین سال)
    "NOx": 0.35,    # g/km — average mixed EU passenger car fleet, TODO verify vs EEA/EMEP Guidebook
    "PM":  0.02,    # g/km — average mixed EU passenger car fleet (exhaust only), TODO verify
}
UNIT_DISTANCE_KM = 1.0
```

### ۴.۴ Validation (اختیاری ولی خیلی ارزش‌مند) — دیتاست `bor_inventaire_polluants_1`

این دیتاست (Atmo NA) مقدار **رسمی سالانه‌ی** انتشار NOx/PM10/PM2.5 و غیره را به‌تفکیک **بخش**
(از جمله بخش حمل‌ونقل) برای کل Bordeaux Métropole دارد. پیشنهاد:

1. بعد از محاسبه‌ی تخمین NOx/PM مدل PoC برای یک منطقه در یک بازه‌ی زمانی،
2. آن را با سهم بخش «حمل‌ونقل» در `bor_inventaire_polluants_1` (مقیاس‌شده به همان بازه‌ی زمانی/جمعیت منطقه) مقایسه کن،
3. در Dashboard یک جمله نشان بده مثل: *"تخمین مدل ما Xx تن NOx است؛ سهم بخش حمل‌ونقل در inventory
   رسمی Atmo NA برای کل متروپل Yx تن در سال است — همان مرتبه‌ی بزرگی (order of magnitude)."*

این گام validation **اختیاری** است (اگر وقت اضافه بود در فاز ۲ اضافه شود)، ولی اگر انجام شود
دقیقاً همان چیزی است که یک هیئت داوری کارآموزی برای «قابل دفاع بودن» PoC می‌خواهد ببیند.

## ۵. کجای پروژه اضافه شود (بدون تغییر اساسی ساختار)

### فایل جدید ۱: `emissions/emission_factors.py`
جدول ثابت بالا + یک تابع کوچک:
```python
def estimate_emissions(vehicle_count: float) -> dict:
    """Returns {"CO2": grams, "NOx": grams, "PM": grams} for one vehicle-count value."""
```

### فایل جدید ۲: `emissions/emissions_engine.py`
یک تابع که یک CSV ترافیک موجود (همان فایل‌های `data/traffic/car_traffic_*.csv`) را می‌خواند
و DataFrame خروجی می‌دهد: `[sensor_id, date, vehicle_count, CO2_g, NOx_g, PM_g]`.
منطق: فقط `pandas`، بدون فایل جدید دانلود از API (از داده‌ی موجود روی دیسک استفاده می‌کند).

### تغییر در `app_pages/city_dashboard.py`
داخل دیکشنری `MODES`، یک ورودی جدید اضافه شود (نه جایگزینی چیزی):
```python
"emission": {
    "label": "Emission Estimate", "icon": "🌫️", "kind": "emission",
    ...
},
```
و یک تابع رندر کوچک جدید (`_render_emission_section`) که:
1. آخرین CSV ترافیک موجود برای همان zone را می‌خواند (از همان تابع `_latest_csv` موجود استفاده می‌کند — تکرار نکن)
2. از `emissions_engine.py` صدا می‌زند
3. نتیجه را با یک نمودار ساده (bar chart یا metric cards: مجموع CO2/NOx/PM برای بازه‌ی انتخاب‌شده) نمایش می‌دهد
4. یک هشدار ثابت نمایش می‌دهد: *"این یک برآورد Rule-Based ساده (Proof of Concept) است، نه یک مدل دقیق فیزیکی یا کالیبره‌شده. فرض‌ها: سرعت ثابت، ناوگان میانگین اروپایی، فاصله‌ی استاندارد ۱ کیلومتر به‌ازای هر عبور."*

**هیچ فایل دیگری نباید تغییر کند.**
